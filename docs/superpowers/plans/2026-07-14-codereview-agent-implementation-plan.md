# CodeReview Agent 实施计划

**依据：** `docs/superpowers/specs/2026-07-14-codereview-agent-design.md`

## 实施原则

- 先建立可运行的受引导 CLI 与确定性项目关系图，再接入大模型审查。
- 大模型是主要问题发现者；本地规则只用于事实提取、风险排序、少量明确风险提示和结果校验。
- 每一阶段均有可运行验证，避免先堆积大量未验证模块。
- 首版只支持 Spring Boot、MyBatis/MySQL、Nacos 与 Vue 项目；不实现自动改码。

## 阶段 1：工程骨架与首次配置

1. 创建 Python 包、CLI 入口、依赖声明、默认配置模板和基础目录结构。
2. 通过 `codereview` 进入受引导会话；实现当前目录、子目录选择、范围说明和退出。
3. 实现首次配置向导：选择 DeepSeek、Ollama 或无模型辅助检查；保存用户配置并限制配置文件权限。
4. 实现 `codereview config`，用于查看、修改 API Key、模型名、Base URL 和默认模式。
5. 添加配置加载失败、无 Key、Ollama 未运行、目录不存在等明确终端提示。

**验证：** 在空目录、普通目录和嵌套目录中运行 CLI；首次配置后再次启动不重复询问；`codereview config` 修改默认模式后生效。

## 阶段 2：项目扫描与事实关系图

1. 递归扫描项目文件，排除构建产物、依赖目录、二进制文件和超大文件。
2. 识别 Java、Vue、JavaScript/TypeScript、SQL、YAML 与常见 Spring Boot、Nacos、MyBatis 标志。
3. 提取 Spring Controller 路由、Service/Mapper 注入、方法调用线索、DTO/Entity、事务注解。
4. 提取 MyBatis Mapper、XML/注解 SQL、SQL 表名与关键操作类型。
5. 提取 Vue Router、组件、Axios/API 调用、接口地址和鉴权相关线索。
6. 提取 YAML/Profile/Nacos 配置、数据源和敏感键线索。
7. 以稳定的数据模型保存“文件事实”和“关系边”，并输出简洁项目地图摘要。

**验证：** 为每类关系编写小型夹具；断言路由、调用、SQL、前端 API 与配置线索被正确提取，且不存在上述技术栈时会明确降级。

## 阶段 3：审查计划与上下文切片

1. 根据关系图生成按风险排序的审查任务：外部入口、写操作、权限、数据库、配置、前端边界、结构性能。
2. 为每个任务构建最小上下文包：目标代码、必要上游/下游片段、关系摘要、任务类型与对应检查清单。
3. 实现文件及任务大小限制，超限时按功能切片分割；记录未覆盖内容。
4. 计算任务输入 Token 估算和输出上限，在线模型模式下在执行前请求用户确认。
5. 增加确定性辅助规则，为风险排序提供线索：敏感键、SQL 拼接、`${}`、写操作、`v-html`、Controller 直连 Mapper 等；这些规则不直接替代模型的深度结论。

**验证：** 对预制项目断言高风险入口优先、每个任务包含正确关联文件、代码过大时有切片与覆盖声明、Token 估算可累计。

## 阶段 4：模型适配与受约束审查

1. 定义统一的模型客户端协议：配置校验、连通性检查、请求、流式/非流式响应、用量统计和规范化错误。
2. 实现 DeepSeek 客户端，使用可配置的 OpenAI 兼容地址、模型名和结构化 JSON 输出。
3. 实现 Ollama 客户端，检查本地服务、列出/验证模型并以相同内部结构返回结果。
4. 为每个审查域实现固定提示词模板：角色、范围、证据要求、触发路径、分级规则、不确定性规则和 JSON 模式。
5. 解析与校验模型 JSON；格式错误时有限重试，仍失败则保留任务失败原因而非伪造结果。
6. 实时显示已完成任务、当前审查域与实际累计 Token。

**验证：** 使用模拟客户端验证请求上下文、提示词边界、JSON 解析、重试、用量累计和 DeepSeek/Ollama 错误提示；真实 API 连通测试仅在用户提供凭据后手动执行。

## 阶段 5：结果治理与严重问题复核

1. 定义统一问题模型：类别、等级、证据位置、触发路径、影响、修复方向、置信度和人工确认标志。
2. 基于类别、文件位置和调用路径生成问题指纹，合并相同根因的重复结论。
3. 校验每个问题是否具有必要证据；缺失证据时降为人工确认或丢弃。
4. 对严重问题创建独立复核任务，只允许“成立 / 不成立 / 证据不足”三种结果。
5. 按复核结果保留、删除或降级严重结论；统一为严重、中等、轻度、优化建议四档。

**验证：** 用模拟初审与复核结果覆盖严重问题的保留、删除、降级；验证同一根因跨 Controller/Service/Mapper 的去重；验证无证据结果不以 Bug 形式出现。

## 阶段 6：终端结果与可选 Markdown 导出

1. 使用清晰的终端布局显示项目摘要、实际覆盖范围、模型状态、实际 Token、四档问题和人工确认项。
2. 审查完成后询问是否导出 Markdown；默认不写文件。
3. Markdown 包含详细问题证据、修复建议、未覆盖范围、修复优先级与下游 Coding Agent 修复交接说明。
4. 对报告中的 API Key 和其他配置凭据执行脱敏；仅在用户选择时写入当前目录下的输出文件。

**验证：** 快照测试终端文本；验证用户拒绝导出时不产生文件；验证导出内容完整、顺序稳定且不含 API Key。

## 阶段 7：示例项目、端到端验证与文档

1. 创建最小示例项目，预埋 SQL 注入、越权、参数校验缺失、事务缺失、N+1、明文密钥、`v-html`、接口契约不一致及正常对照案例。
2. 执行无模型辅助检查、模拟模型审查、模型失败、严重问题复核和 Markdown 导出的端到端测试。
3. 编写 README：Python 安装、首次配置、DeepSeek/Ollama 连接、审查流程、分级含义、隐私说明和限制。
4. 对真实 DeepSeek 与本地 Ollama 各完成一次人工冒烟验证，并记录不上传真实 API Key 的示例输出。

**验收标准：** 用户在目标目录运行 `codereview` 后能按提示完成审查；Agent 能正确识别目标技术栈，按功能切片执行受约束的模型审查，显示并可导出带证据和复核状态的四档结论；模型不可用时明确告知并保留已完成结果；工具不修改被审查项目代码。
