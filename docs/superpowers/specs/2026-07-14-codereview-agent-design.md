# CodeReview Agent 设计规格

## 1. 定位与目标

CodeReview Agent 是面向 **Spring Boot + MySQL + Nacos + Vue** 课程设计项目的引导式命令行代码审查工具。

它的核心不是替代大模型发现问题，而是为大模型建立可信的项目上下文、限定技术栈审查边界、强制代码证据、统一风险分级，并对严重结论进行二次复核。用户可以用较低成本得到一份有优先级、可交接给其他 Coding Agent 执行修改的详细审查结果。

首版只审查代码，不自动修改代码。

## 2. 非目标

- 不提供 Web 或图形界面。
- 不自动修改源代码、创建补丁或执行数据库迁移。
- 不承诺替代人工安全审计。
- 不以任意编程语言或任意框架为支持范围；首版聚焦已定义技术栈。

## 3. 用户交互

用户在待审查项目的根目录执行 `codereview`，进入受引导的终端会话。

```text
CodeReview Agent
当前工作目录：/path/to/project

1. 审查当前目录
2. 选择当前目录下的子目录
3. 查看当前模型与审查范围
4. 查看审查范围与分级说明
0. 退出
```

首次运行时，若未找到用户配置，Agent 进入一次性的配置向导。用户选择默认模式并完成必要配置后，配置保存在用户目录的私有配置文件中；后续执行 `codereview` 时直接加载默认模型，不再次要求选择。

配置向导提供三种模式：

1. DeepSeek API：用户提供自己的 API Key 和可配置模型名。
2. 本地 Ollama：代码和模型调用均留在本机，适合敏感项目。
3. 辅助本地检查：没有模型时仅发现少量可确定的规则问题，不能声称完成 AI 深度审查。

用户需要查看、修改 API Key、切换默认模型或改用本地模式时，执行 `codereview config` 进入同样受引导的配置流程。配置文件包含选定提供方、模型名和连接地址；API Key 不显示在终端、日志或报告中，且配置文件必须只允许当前用户读写。

在线模型审查前，Agent 根据计划中的代码分块和每项输出上限估算 Token 区间，展示估算值并请求用户确认。审查结束后显示模型实际返回的累计 Token 用量。

终端必须显示审查过程和问题。审查结束时，用户可选择导出完整 Markdown 审查报告；报告不是课程实验报告，而是本产品的详细交付物。

## 4. 风险分级

| 等级 | 定义 |
| --- | --- |
| 严重 Bug | 可被利用的安全漏洞、敏感数据泄露、数据破坏、核心服务不可用等高影响问题；必须通过二次复核。 |
| 中等 Bug | 在常见场景可能导致功能异常、错误结果、明显性能问题或安全边界缺失。 |
| 轻度 Bug | 低概率边界条件、可维护性问题、规范性问题或潜在风险。 |
| 优化建议 | 目前不一定错误，但可明确改善性能、结构、可读性或资源使用的建议。 |

不能从代码与关联上下文确认的问题必须标记为“需人工确认”，不能断言为 Bug。

## 5. 智能审查引擎（核心竞争力）

### 5.1 事实优先的项目关系图

Agent 先从文件和语法/模式中提取可验证事实，再让模型基于这些事实推理，不能让模型凭空猜测架构。

它应建立以下关系：

```text
Vue 路由/页面
  -> 前端 API 调用
  -> Spring Controller 路由
  -> Service 业务方法
  -> Mapper / XML / 注解 SQL
  -> MySQL 表与字段

Nacos 配置
  -> Spring Profile / 数据源 / 服务发现 / 配置项使用处
```

首版提取的事实包括：

- Spring 的 Controller、路由、方法参数、注入依赖、Service 调用和 Mapper 调用。
- MyBatis Mapper 方法与 XML SQL 或注解 SQL 的对应关系。
- Vue Router、页面组件、Axios/封装 API 方法和请求地址。
- `application.yml`、`bootstrap.yml`、Profile、Nacos 配置、数据源和敏感键。
- DTO、Entity、请求参数和响应对象的关联。

这些事实组成“功能切片”。例如审查一个删除接口时，模型得到的上下文应包括前端调用、路由、Controller、权限线索、Service、事务线索和关联删除 SQL，而不是孤立地审查某一个文件。

### 5.2 风险优先的审查任务编排

Agent 不等量审查所有文件，而是生成带优先级的任务队列：

1. 外部入口：Controller、登录注册、文件上传、管理端接口。
2. 数据写操作：新增、修改、删除、批量操作、状态变更。
3. 身份与权限：JWT、Interceptor、Spring Security、用户 ID 来源。
4. 数据库边界：动态 SQL、Mapper、复杂查询、事务。
5. 配置与服务治理：Nacos、数据源、密钥、环境 Profile。
6. 前端安全边界：路由守卫、Token、Axios、`v-html`、上传。
7. 结构和性能：重复业务逻辑、超长方法、循环查库、缓存与查询优化。

每项任务由目标代码、最小必要关联上下文、技术栈检查清单和固定输出结构组成。代码较大时按功能切片继续拆分，防止上下文超限，同时保留任务与项目关系图的引用。

### 5.3 技术栈审查边界

| 范围 | 必审技术细节 |
| --- | --- |
| Spring Boot 接口层 | 缺少鉴权、IDOR 越权、参数校验、Entity 直接入参导致批量赋值、异常泄露。 |
| Service 业务层 | 多次写库的事务边界和回滚、先查后改的并发风险、状态流转绕过、循环查库、职责过重。 |
| MyBatis / SQL | `${}` 或字符串拼接注入、更新/删除遗漏 `WHERE`、分页缺失、N+1、全表扫描、动态 SQL 条件、结果映射。 |
| MySQL 数据设计 | 金额使用 `float/double`、时间/时区、索引覆盖、唯一约束、软删除一致性。没有 DDL 或执行计划时，索引结论只能标为建议或人工确认。 |
| Nacos / 配置 | 密钥和密码明文、Profile 混用、关键安全配置覆盖、服务地址/命名空间/分组误配风险。Agent 只评估可见配置，不宣称验证运行中集群状态。 |
| Vue / JS | `v-html`、动态 URL、上传的 XSS 风险；Token 注入、过期和 401；接口错误处理；表单校验；异步状态和资源清理。 |
| 跨端契约 | 前端路径、参数名、类型、分页格式与后端接口的一致性；前端权限展示与后端强制鉴权的一致性。仅有前端隐藏按钮时是体验提示，只有后端也缺少保护时才是严重安全问题。 |

### 5.4 模型审查协议

每次调用都带有与任务对应的审查边界，并强制模型遵循：

- 仅检查本次提供代码与上下文能够证明的范围。
- 每个问题必须给出文件、方法或接口、代码证据和触发/调用路径。
- 安全结论必须说明外部输入来源、处理过程和危险落点。
- 无法确认时输出“需人工确认”，不能把推测写成事实。
- 不输出无证据的泛化建议。
- 优化建议必须给出优化对象、预期收益和适用前提。

模型返回结构化 JSON，至少包含：问题类别、严重等级、证据位置、触发路径、影响、修复方向、置信度、是否需要复核。

### 5.5 结果归并与严重问题二次复核

Agent 使用问题类型、文件位置和调用路径构建问题指纹，合并同一根因在 Controller、Service、Mapper 等位置被重复发现的结论。

被初审模型判为“严重 Bug”的问题必须进入独立复核任务。复核输入只包含初审结论、相关代码片段、必要调用链和严重等级标准。复核结果只能是：

- 成立：保留严重等级。
- 不成立：删除该结论。
- 证据不足：降级为中等、轻度或需人工确认。

## 6. 详细结果与下游交接

终端输出必须显示项目摘要、实际审查范围、问题分级、位置、影响和简短建议。Markdown 导出保留更完整的证据、关联关系和修复顺序。

Markdown 的末尾固定包含“下游 Coding Agent 修复交接说明”：按优先级列出待修问题、涉及文件、修改目标、约束及修改后的验证方式。它不生成或应用代码修改。

## 7. 可靠性与隐私

- API Key 无效、网络错误、模型未运行、速率限制、返回 JSON 无法解析时，终端给出明确原因和下一步操作；不得伪造 AI 审查完成。
- 模型调用失败时保留已完成任务，允许继续输出部分结果。
- 代码上下文超限时自动按功能切片拆分，并声明未覆盖的内容。
- API Key 不出现在终端、日志或报告中；保存配置时仅写入用户私有配置文件，并限制为当前用户可读写。
- Ollama 模式不向外部 API 发送代码；DeepSeek 模式需在会话中提示代码会被发送给用户选择的提供方。

## 8. 验收与测试策略

实现阶段应准备一个小型 Spring Boot + MySQL + Nacos + Vue 示例项目，其中预埋并标记以下案例：SQL 注入、越权、缺少参数校验、事务缺失、N+1、明文密钥、`v-html` 风险、前后端接口不匹配，以及正常对照代码。

测试至少覆盖：

- 当前目录和子目录选择。
- 技术栈识别和项目关系图关键关系。
- 审查任务优先级与上下文切片。
- 模型结构化结果的解析、去重与四档分级。
- 严重问题复核后的保留、降级和删除。
- API / Ollama / 无模型模式的失败提示与部分结果。
- 终端输出和可选 Markdown 导出。

## 9. 后续扩展（不纳入首版）

- 接入更多 OpenAI 兼容或 Anthropic 兼容模型提供方。
- 支持 Python、React 和其他后端框架。
- Git 仓库、分支和变更集审查。
- 生成补丁并由用户确认后应用。
